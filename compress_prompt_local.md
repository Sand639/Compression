# 可逆圧縮プログラム ARC1 自律改良タスク（最新版・ローカル実行）

あなたは可逆圧縮プログラム ARC1 を、このローカルマシン上で直接編集して改良し続けるエージェントです。
対話確認 (Pause) は一切せず、ターミナル実行ツールで自走してください。質問で止まらないこと。
このセッションはいつ中断されるか分かりません（利用制限）。
**改善が通ったら必ず即コミット**し、未コミットの作業を長時間残さないこと。

---

## 現状（必ず最初に読むこと）

このプロジェクトは既に高度に最適化済み。直近の到達点:
- **現在の BEST ≈ 1,171,165 B（本物5ファイル, output.enc）。7z(1,640,836) を約28.6%下回る。round-trip 5/5 完全一致。**
- 全5ファイルが CM をバックエンドに採用。現在の各ファイルの勝者:

  | ファイル | 圧縮後(目安) | 採用方式 |
  |---|---|---|
  | TeraPad.exe | 約426,631 | BCJ(LZMA SDK)+CM |
  | explosion.wav | 約230,887 | WAV(M/S選択+ブロックLPC+sign-sign LMS)+CM |
  | wagahaiwa_nekodearu.txt | 約227,372 | CM |

- **状態は「ほぼ頭打ち」**。APM段数・ミキサー文脈サイズ・学習率・stride・テーブルサイズの
  微調整は既にほぼ掘り尽くし、得られても数バイト〜数十バイト、または失敗が多い。
  **微調整に深入りしないこと。** 残る伸びしろは下の「大型レバー」のみ。

## 絶対ルール（破ったら即 revert）

- 使用言語は C/C++ のみ（C++17/20）。**外部ライブラリ禁止**、標準ライブラリのみ。
- 圧縮は**完全可逆**。対象5ファイルを1バイトの狂いもなく復元できること。
- **実行時間の制限はなし**。遅くてもよいので「小ささ」を最優先する。
- 既存のセルフテスト (`RunSelfTests`) が常に PASS すること。
- フォーマット（アーカイブ/ストリーム）を変えたら**必ずバージョン番号を上げ**、旧復号と混ざらないようにする。
- **メモリに注意**: 既に exe プロファイルは大きなコンテキストテーブル(TBITS28)を使い、
  TBITS29 で並列ハーネスがメモリ超過した実績がある。peak memory を増やす変更は慎重に。
  まず measure.exe でスクリーニングし、OOM やスワップで遅くなる変更は避ける。

## 起動時にやること（順番厳守）

1. `git status` / `git log --oneline -8` で現在の HEAD を確認。未コミット変更があれば、
   まず現状を `session-start` としてコミットする。
2. **本物5ファイルの存在とサイズを検証**（壊れたデータでの改良を防ぐ）:
   | ファイル | 期待サイズ(bytes) |
   |---|---|
   | wagahaiwa_nekodearu.txt | 749,051 |
   | yuuki_256.bmp | 641,076 |
   | TeraPad.exe | 1,462,272 |
   | explosion.wav | 599,084 |
   | hal.bmp | 712,922 |
   - どれか欠落/サイズ不一致なら、ここで停止し原因（破損やgitignore除外の疑い）を報告して指示を待つ。
     特に過去に Linux 実験で TeraPad.exe を gitignore 除外した経緯があるので要確認。
3. `PROGRESS_COMPRESS.md` と `LEDGER.md` を読む（過去の採用・却下・計測手順を把握するため）。
4. **真の BEST を実測で確定する**（古い数値や4ファイル実験を信用しない）:
   - PROGRESS_COMPRESS.md に記録された Windows/MSVC ビルド手順
     （VsDevCmd 経由、`cl /nologo /std:c++20 /O2 /EHsc /utf-8`、**/utf-8 必須**）でビルド。
     プロジェクトが単一ファイル(bwt.cpp)か分割モジュールかはディスク上の実体を見て判断し、それに合わせる。
   - `bwt.exe` を実行し、本物5ファイルで output.enc を生成→復元→5/5バイト一致を確認。
     この output.enc のサイズを **BEST** とし、PROGRESS_COMPRESS.md に「session-start BEST」として明記。
   - 高速スクリーニング用に `measure.exe`（約3秒）も同手順でビルドして使えるようにする。

## 計測基準（唯一の真実）

- **スコア = bwt.exe が出す output.enc の総バイト数**（5ファイルを1つにまとめた合計）。個別ファイルでは判断しない。
- 開発中の高速判定は `measure.exe`（3秒, payload合計+ヘッダ）でスクリーニングしてよいが、
  **合否の最終確定は必ず `bwt.exe`** で行う（可逆性ゲートを兼ねる）。

## 1イテレーションの手順

1. `LEDGER.md` を読み、下の「大型レバー」から未着手で最も有望なものを1つ選ぶ。
   「何を・なぜ・どう変えるか」を1〜2行 LEDGER.md に追記してから着手する。
2. 変更を実装する。関数シグネチャを変えたら `compress.h`（または該当ヘッダ）も更新する。
3. `measure.exe` で素早くスクリーニング → 縮みそうなら `bwt.exe` で本計測。
4. **合否（3つ同時成立で合格）**:
   - (a) 完全可逆（全5ファイルがバイト一致）
   - (b) セルフテスト PASS
   - (c) 新スコア **<** 現在の BEST（**厳密に小さい**。同値は不合格）
5. **合格** → `git commit`（メッセージに「手法名 / 旧BEST→新BEST / 削減量」）。
   BEST 更新、LEDGER.md と PROGRESS_COMPRESS.md に成功と削減量を記録。
6. **不合格（縮まない/テスト失敗）** →
   - まず LEDGER.md と PROGRESS_COMPRESS.md に「何を試し、なぜ駄目か（何バイト増えたか等）」を詳細に追記。
   - 次にコード変更だけを戻す: `git restore "*.cpp" "*.h"`（.md のメモは戻さない。メモは絶対に restore しない）。
   - メモだけを `git add LEDGER.md PROGRESS_COMPRESS.md` して
     `git commit -m "log: failed attempt (手法名)"` でコミットし、失敗の痕跡を残す。
   - コンパイルエラー/可逆性破綻の場合のみ、原因分析して**最大2回**まで自動修正を試してよい。深追いはしない。
7. 次のイテレーションへ。

## 改善アイデアの引き出し ── 残った「大型レバー」（微調整は原則やらない）

微調整は掘り尽くしているので、効果が大きく未着手の構造的変更を優先する。
各レバーは可逆性を最優先で検証し、必ず合否ゲートを通すこと。

1. **CM 確率カウンタを「ビット履歴ステートマシン + StateMap」へ置換／併用**
   現状は (prob<<4)|count の単純適応カウンタ。lpaq/zpaq 系のように「直近ビット履歴の状態」を持ち、
   状態→確率を StateMap で学習する方式は CM 系で最大級のレバー。全ファイルに効く可能性。
   ※ match モデルの StateMap 化は過去に失敗したが、これは**メイン文脈カウンタ側**の話で別物。慎重に導入。

2. **間接文脈モデル（ICM）を1〜2本追加**
   (文脈 → その文脈で最近見たバイト履歴 → StateMap) という間接参照モデル。paq/zpaq の主力の一つ。

3. **テキスト専用モデル（wagahaiwa.txt 狙い）**
   - 単語ハッシュ文脈（word model）: 現在の単語の途中までをハッシュして文脈にする。自然言語に強い。
   - **Shift-JIS 2バイト構造を意識した文脈**: リードバイト/トレイルバイトの位置や直前リードバイトを
     文脈に加える。日本語テキストは2バイト単位の規則性があり、バイト単位文脈だけでは取りこぼす。

4. **画像 CALIC 級の文脈モデリング（hal.bmp 狙い）**
   MED/GAP 予測の残差を、勾配・誤差エネルギーで量子化した文脈で符号化し、
   文脈別のバイアス補正（error feedback / context-conditioned bias）を入れる。
   現状の「予測残差→CM」より一段強い、自然画像向けの王道。

5. **音声の高次/多段適応予測（explosion.wav 狙い）**
   現状は ブロックLPC + 単段 sign-sign LMS。NLMS（正規化LMS）や多段カスケード、
   より長いタップの適応フィルタで残差を白色化する。Monkey's Audio / OptimFROG 系の発想。
   （sign-sign 単段が最良だった経緯があるので、NLMS は別物として丁寧に。全演算16bit丸めで可逆厳守。）

6. **実行ファイル専用の文脈（TeraPad.exe 狙い）**
   命令内バイト位置やオペコード/オペランドを区別する文脈、ModRM/相対オフセット境界を意識した文脈。
   （Jcc 拡張BCJ は偽陽性で失敗済みなので、BCJ 拡張ではなく CM 側の文脈設計で攻める。）

7. **（最後の手段）微調整**
   1〜6が尽きたら、テーブル/学習率/APMの未試験の組み合わせを measure.exe で素早く探索。
   ただし逓減が激しいので、ここに入ったら停止条件に従って早めに切り上げる。

（これら以外に思いついた可逆な改善を試してもよい。必ず合否ゲートを通すこと。）
